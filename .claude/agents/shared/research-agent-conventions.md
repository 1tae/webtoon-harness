# 리서치팀 공통 규약

> **이 문서는 리서치팀 4인 조사자(trend-scout, platform-ranker, audience-analyst, hook-analyst)가 공유하는 공통 작업 규약이다.** trend-synthesizer는 종합 역할이므로 별도 규약을 따른다.

## 공통 사용 스킬
- `webtoon-trend-research` — 조사 대상, 신뢰 출처 우선순위, 관찰 프레임, 출력 구조를 이 스킬에서 따른다. **착수 전 반드시 로드한다.**

## 공통 입력
- `_workspace/00_input/brief.md` — 사용자 입력·회차 번호·제약. 타깃 장르/톤 힌트가 있으면 우선 반영한다.

## 공통 출력 경로 패턴
- 출력 위치: `_workspace/01_research/{에이전트 이름}.md`
- 형식: 마크다운. 각 보고서는 자기 섹션 구조(`요약` → `본문 세부` → `기획 시사점` → `출처`)를 갖추고, 끝에 출처를 모은다.

## 공통 재호출 지침 (후속 작업)
- 자신의 출력 파일이 이미 있으면 **Read하여 변경된 신호만 갱신한다.** 전체를 재작성하지 않는다.
- 사용자 피드백이 있으면 해당 섹션만 보강한다.
- 랭킹·트렌드는 관측 일자를 갱신하고, 라이프사이클 단계를 업데이트한다.

## 공통 에러 핸들링
- WebSearch/WebFetch 결과가 빈약하면 검색어를 한국어·영어·플랫폼명·작품명으로 바꿔 재시도한다(최대 3회).
- 접근 불가 출처는 URL과 실패 사유를 남겨 후속 재시도가 가능하게 한다.
- 2차 출처(뉴스·집계 블로그·공식 발표·커뮤니티 요약)로 보완하고, 출처 한계를 명시한다.
- 추측을 사실처럼 적지 않는다. 불확실은 "추정"으로 라벨링한다.
- 데이터 부족 영역은 비워두지 말고 "확인 불가"로 명시해 후속 재조사를 가능하게 한다.
- 모든 데이터 소스 접근이 실패하면 trend-synthesizer와 오케스트레이터에 에스컬레이션한다. 빈 보고서를 조용히 제출하지 않는다.

## 공통 협업 구조
- 리서치팀 4인 조사자 중 하나로, 다른 세 명과 병렬로 일한다.
- 조사 중 흥미로운 발견을 SendMessage로 교차 공유한다.
- 산출물은 **trend-synthesizer**가 4개 조사 결과를 종합할 때의 핵심 입력이다.
- 최종 종합본 `trend-brief.md`는 시나리오팀(concept-architect·series-plotter·twist-master)이 소비하므로, 발견은 **"이야기로 전환 가능한 형태"**(트로프·기법·구조)로 남긴다.

## 공통 출처 규칙
- 모든 항목에 **URL과 관측 일자**를 붙인다.
- **상충 정보는 양쪽 병기**한다. 한쪽을 임의 삭제하지 않는다.
- 검증 안 된 효과·수치는 "추정" 또는 "가설"로 라벨링한다.
